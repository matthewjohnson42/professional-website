MACCS Neural Network

The MACCS Neural Network was a research project that I conducted in the summer of 2017 at Augsburg College (now University). The goal of the project was to automate the recognition of anomalous activity in the data of the MACCS project at Augsburg. The MACCS project is a study of the earth's magnetosphere near the north pole. Activity recorded by the MACCS projects included things such as solar flares. The final version of the neural network recognized the so termed anomalous patterns in magnetosphere data with a peak accuracy of 97.16% and an average accuracy for that particular neural network architecture of 96.58%.

The first step of the project was to create a labeled set of magnetosphere data. It was determined that there was an insufficient amount of recent data from the MACCS project, a result of hardware errors on the servers that stored the data. Instead, we utilized data from the British Antarctic Survey's search coil magnetometers located at the Halley research station. The data was parsed by myself and a team member of the Augsburg College Physics Department, Jennifer Posch, and a formal definition for anomalies was created. This definition was then used to hand label a set of 1117 days worth of data from the search coil magnetometers at BAS Halley Site.

After having labeled data from the BAS Halley Site's SCM array, the raw data that the labels corresponded to were passed through a fast fourier transform using the SciPy library for the python programming language. After the data was transformed, it was used as input to neural networks that were defined using the Keras python library with the Keras code being translated by the library into TensorFlow code. Many different network architectures were tried, all of them closely following the pioneering convolutional neural network architecture defined by LeCun et al in 1998 (referred to as LeNet). These networks were trained on an Intel 32-core server with varying training parameters.

There were several difficulties that were encountered when training networks. One was that the networks seemed to be overfitting their models to the data, meaning that the networks were being trained to recognize patterns that while present were not relevant to the pattern that the labels were intended to correspond to. This was determined by conversation with the mentoring professor, Erik Steinmetz, where it was discussed that neural networks do very well with classification tasks that have sufficiently distinct inputs and yet the networks that were being trained were not doing well. The overfitting hypothesis was confirmed when dropout layers were added to the network, a training parameter for the Keras library that drops a certain percentage of the model each time the model is fit to a particular set of the training data. Another difficulty that was encountered was the need to format the data to a minimally precise representation of the data, which would then allow for the network to more reliably extract the patterns from the data. This problem was similar to the over fitting problem, where a more precise representation of the data allowed for the network to be fit to patterns in that were present in smaller variances in the data but were not relevant to the pattern that was intended to be recognized. This was solved by splitting the data into ranges of fixed size, and by removing the lowest frequency from those ranges to remove some of the noise from the data. With these changes, the network's accuracy jumped from ~85% to ~96%.

I also created a research poster and wrote a paper about this research. The poster may be found here. The paper may be found here.

Pascal Compiler

Another project that I undertook is my Pascal compiler, which was a project that constituted the capstone for the Computer Science curriculum at Augsburg College, now Augsburg University. The compiler is implemented in Java and was tested using Junit. It was completed in April 2017.

The compiler is a recursive descent compiler that utilizes a non deterministic finite automaton for tokenization and a context free grammar for the construction of the syntax tree. The target language is MIPS assembly, an older assembly language that is utilized in coursework at Augsburg University. The grammar was provided to the students along with the recursive descent compiler architecture. Tokenization was to be completed using the tool JFlex, a piece of Java software that takes as input a text file specifying a state machine and providing as output the Java implementation of that state machine. Implementation and unit testing the compiler specifications were left to the students. Put simply, this compiler takes as input a file containing text written in a subset of the language Pascal and provides as output the assembly instructions for a CPU that uses the MIPS architecture.

There are several things that are notable about this project in regards to my performance on implementation. First, the compiler performs exactly to specification. I received a 4.0 of 4.0 for this capstone. Second, the compiler included functionality that was dismissed from the compiler specifications based on the amount of time remaining for students to be able to complete the project. This functionality was the inclusion of callable methods/functions and of arrays.

I very much enjoyed the compiler project. It was a reasonably complex piece of software and although it does not compile all Pascal code, I am content to know that I was capable of understanding compiler theory with sufficient precision so as to be able to create this compiler. The theory really was what interested me most about the course as it really does dig into the logic that it was necessary to develop in order to be able to create compilers. This is an example of my preference for the theory and concepts involved in a project above the details of an implementation.

Saberpack Production Control

This is a project that I undertook in months of May and June in 2016. It is a piece of software that tracks the production of a glass packaging product for the purposes of retaining quality assurance for that glass packaging product. The product is called glass interleaving powder, and is a substance placed between large flat panes of glass prior to the glass being shipped from the glass production facility. The software takes as input the weight of a bag of the interleaving powder and provides as output a bag label with a lot and bag number. This data, along with user input specifying environmental conditions such as temperature and relative humidity, was stored in a database for later reference. The software was deployed to the manufacturing environments at Saberpack in June of 2016, where it continues to reliably track the production of material.

This project was a rebuild of an existing piece of software that was written in Visual Basic. The existing piece of software was written in 2008 and was desgined to run on Windows XP machines and utilizing the .NET framework version 2.0 to interface with comm port peripherals and with a SQL Server Express 2007 instance. The goal of this project was to get the software running on new machines that could replace the aging equipment located in an environment with high amounts of particulate in the air.

There were major hurdles in this project, with the first one being a .NET dependency that did not have an equivalent distribution for versions of Windows later than XP. Given that XP was no longer being supported by Microsoft, it was suggested that the code base be updated to run on a more modern operating system. Windows 10 was chosen as the operating system based on Microsoft's stated plans for supporting their operating systems, and C# was chosen as the language of implementation based on the observation that the language is well supported on modern Windows distributions and has well supported access to the .NET framework. Another major hurdle for myself as a developer was the schedule for the delivery of this software. The planned expense for this project was fairly low and thus the time I had to develop this software was limited. The result this was that corners were cut in the course of development to ensure the project was completed on time. For example, there was no test suite used to test this software; all testing was done by analysing stack traces or using break points and console output.

There is one main thing that is notable about this project in regrads to my performance: I went into a situation where I had no knowledge of the existing tech stack, limited documentation (only code comments), no communication with the original developer, and yet within two months I had learned enough about the function of the software to be able to provide a technology solution for the company. I did not know SQL Server before this project. I did not know C# prior to this project. I had never worked with the .NET framework. I implemented this software solution using C#.NET and SQL Server Express in two months.

Sentiment Analysis Self Study

I have an ongoing interest in sentiment analysis. Sentiment analysis as it is referred to in some academic publications is the study of affect, emotion, or sentiment in a given portion of text. It is an emerging area of study. One application of sentiment analysis that was in the news in early 2018 was Facebook's determination to ramp up efforts to detect hate speech and inaccurate news reports. While the analysis that is likely to be relevant to the automated detection of inaccuracy in reporting is not quite the same area of analysis as sentiment analysis, the analysis that might be utilized in the pursuit of an automated detection of inflammatory speech is quite close to sentiment analysis if not the same. My interest in sentiment analysis rose in part from an observation of language forming a sort of natural barrier to human knowledge, if not only in that knowledge which may be meaningfully expressed. I chose to pursue a study of sentiment analysis during my last semester at Augsburg University under Professor Erik Steinmetz. The product of that study was a paper that can be found here.

I have created a fairly small repository of the programming that I've done on my own time relating to sentiment analysis. I haven't done much with it yet, but should I happen to find for myself that I am capable of pursuing a continued study of sentiment analysis in my spare time I might consider making something more of it. The first step for this ongoing project is to create a very simple sentiment analyzer that takes in a string of text, splits it into distinct words and then assigns a score to each word based on a sentiment lexicon (I am utilizing SentiWordNet 3.0 on my first go). The score of all words in the document are then totaled, giving a very rough approximation of the general sentiment of the text either positive or negative. This is the simplest form of sentiment analysis and is referred to as polarity classification.
